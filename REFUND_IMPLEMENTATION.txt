## 📊 DATABASE SCHEMA

CREATE TABLE refunds (
  id                      BIGINT PRIMARY KEY AUTO_INCREMENT
  staff_user_id          BIGINT FK → users (who issued)
  student_user_id        BIGINT FK → users (who received)
  amount                 DECIMAL(12,2) (how much)
  reason                 VARCHAR(255) (why)
  related_transaction_type VARCHAR(255) (e.g., WalletLoadLog)
  related_transaction_id BIGINT (link to original transaction)
  refunded_at            TIMESTAMP (when processed)
  created_at             TIMESTAMP
  updated_at             TIMESTAMP

  INDEXES:
  - staff_user_id + created_at
  - student_user_id + created_at
  - related_transaction_type + related_transaction_id
)

## 🔒 SECURITY FEATURES

✓ ROLE-BASED ACCESS
  • Only staff (role='staff') can issue refunds
  • Students can only view their own refunds
  • Admins can view all refunds

✓ VALIDATION
  • Amount must be > 0
  • Student ID must exist
  • Reason required (no empty refunds)

✓ ATOMIC TRANSACTIONS
  • Refund creation + wallet update happen together
  • If wallet update fails, entire transaction rolls back
  • No orphaned refund records

✓ AUDIT TRAIL
  • All refunds recorded permanently
  • Staff member ID linked to each refund
  • Timestamps for when refund was processed
  • Optional link to original transaction

## 📝 REAL EXAMPLE

SCENARIO: Student reports being charged ₱500 twice for wallet load

STAFF ACTION:
POST /staff/refunds
{
  "student_user_id": 123,
  "amount": 500.00,
  "reason": "Refunding duplicate wallet load transaction",
  "related_transaction_type": "WalletLoadLog",
  "related_transaction_id": 789
}

SYSTEM PROCESSES:
  1. Validates staff role ✓
  2. Validates amount > 0 ✓
  3. Validates student exists ✓
  4. Creates refund record:
     - staff_user_id: 2 (logged-in staff)
     - student_user_id: 123
     - amount: 500.00
     - reason: "Refunding duplicate..."
     - related_transaction_type: "WalletLoadLog"
     - related_transaction_id: 789
     - refunded_at: 2026-05-04 23:26:29
  5. Updates student wallet_balance += 500.00
  6. Returns confirmation:
     {
       "message": "Refund processed successfully",
       "refund": { ... },
       "student_new_balance": 1500.00
     }

TRACKING:
  • Student sees refund in GET /student/refunds/history
  • Staff sees refund in GET /staff/refunds/history
  • Admin sees refund in GET /admin/refunds
  • All queries show who refunded, how much, why, and when
